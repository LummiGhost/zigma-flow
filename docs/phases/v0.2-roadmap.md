# Zigma Flow v0.2 — Autonomous Execution Hardening

## 0. 一句话定位

v0.1 让 workflow 能跑通；v0.2 让 workflow **跑得稳、跑得快、跑得住**。

## 1. v0.1 收敛点回顾

- Engine 是唯一状态推进者（MVP contract §2.3）已落地。
- Script、Check、Router、Signal、Retry、Optional Job、Artifact、Event 全部可用，464+ 测试在 CI 绿。
- PR #87 引入第一个自动化 Agent backend（Claude Code）和 `run-all` 命令，让端到端自动执行成为可能。

v0.1 的"半自动 Agent"假设（用户手动复制 prompt 给 Claude Code）至此被自动 backend 取代。**v0.2 的所有改动都基于"自动 backend 是首选路径"**。

## 2. v0.2 痛点诊断

| # | 痛点 | 现象 | 影响 |
|---|---|---|---|
| D-1 | run-all 失败立刻终止整个 run | Claude CLI 超时/退出非零 → 立刻 `step_failed` → `run.failed`，不走 `retry.max_attempts` | 失去 MVP §FR-012 的 retry 语义 |
| D-2 | agent 进程产物丢失 | stdout/stderr 仅截尾 1000 字符塞进错误消息，不落 artifact | 失败无法复盘，违反 PRD §13 artifact 一等公民原则 |
| D-3 | run-all 无 observability | 只有 `prompt_generated` 和 `step_failed`，没有 `agent_invoked`/`agent_completed` 事件 | event log 缺位，违反 mvp-contracts §2.4 |
| D-4 | 顺序执行造成 wall-clock 浪费 | `code-map` + `risk-scan` 都是 read-only，被串行 | dogfood 一次 30+ 分钟 |
| D-5 | 无人工介入点 | 想在 review 前暂停只能 `abort`，丢失上下文 | PR 集成、合并审批无落点 |
| D-6 | run-all 不支持续跑 | 每次都创建新 run；中途崩了无法接着跑 | 长 run 风险放大 |
| D-7 | Ctrl-C 行为不定义 | execa 子进程是否清理、events 是否补写 `run_cancelled` 未规范 | 留下脏 run |

## 3. v0.2 阶段切分

| 阶段 | 名称 | 主要解决 | PR 数 | 估算 |
|---|---|---|---|---|
| P13 | Agent Adapter Hardening | D-1, D-2, D-3, D-6, D-7 | 1 | 1–1.5 周 |
| P14 | Concurrent Read-Only Execution | D-4 | 1 | 1 周 |
| P15 | Human Gate Step | D-5 | 1 | 1 周 |

顺序原则：

1. **P13 先做**：P14 和 P15 都会引入新事件类型，必须先把 agent 调用路径的事件契约冻结。
2. **P14 在 P13 之后**：并发只会放大 adapter 缺陷；先稳后并发。
3. **P15 在最后**：human gate 涉及 CLI 命令面扩展，独立于前两项，可与 P14 并行评估，但建议保持顺序避免冲突。

## 4. 跨阶段公共约束

以下原则在 P13–P15 中必须一致遵守，落到每阶段 Step 1 用例文档中：

### 4.1 Engine 唯一写者原则（不变）

- run-all 的主循环可以拆分，但所有 state.json 写入仍走 `LocalStateStore.writeSnapshot`。
- 任何"agent backend 内部触发的状态变化"都必须翻译为 Engine 入口调用（`acceptAgentReport`、新增的 `recordAgentFailure` 等）。

### 4.2 事件 ID 不再用 parseInt hack

- 现状 `run-all.ts` 与 `engine/index.ts:419` 反复使用 `parseInt(lastId.replace("evt-", ""), 10)` 推下一个事件号。
- v0.2 必须在 `src/events/` 暴露 `nextSequentialEventId(runDir): Promise<string>` 端口。所有写事件的地方（engine、run-all、未来 backend wrapper）统一走它。

### 4.3 事件契约扩张

| 新增事件类型 | 阶段 | 用途 |
|---|---|---|
| `agent_invoked` | P13 | backend 启动子进程时，记录 backend 名称、命令、超时 |
| `agent_completed` | P13 | backend 成功且 report.json 合法时 |
| `agent_timed_out` | P13 | backend 触发 timeout 时 |
| `agent_failed` | P13 | backend 非零退出或 report.json 缺失/非法时 |
| `agent_cancelled` | P13 | Ctrl-C 或 AbortSignal 触发时 |
| `run_cancelled` | P13 | run 进入 cancelled 终态时（补齐 MVP contracts §2.4 列表中已声明但未实现的事件）|
| `human_gate_waiting` | P15 | 进入 human step 时 |
| `human_decision` | P15 | 收到 approve/reject 时 |

### 4.4 Artifact kind 扩张

| 新 artifact kind | 阶段 | 内容 |
|---|---|---|
| `agent_stdout` | P13 | backend 子进程 stdout 完整内容 |
| `agent_stderr` | P13 | backend 子进程 stderr 完整内容 |
| `agent_invocation` | P13 | backend 调用元信息（command, args, env keys, timeout, exit_code, duration）|
| `human_decision_record` | P15 | 决策时间、决策者、comment、是否触发 retry |

### 4.5 MVP Out-of-Scope 守门

v0.2 三阶段**不引入**以下能力（仍属 PRD §24 的 v0.3+ 范围）：

- Git worktree per writable job（D-4 在 v0.2 只解决 **read-only** 并发）
- 嵌套 workflow / workflow step
- PR 集成、自动 commit
- 远程 Skill Registry、MCP、Docker
- 完整 event sourcing 重建（events.jsonl → state.json 回放）
- 自动多 Agent 调度（v0.2 的并发是任务级并发，仍只允许 1 个 agent 在写 job 中）

如果实现阶段发现某项必要，必须先回到 PRD/architecture 修订再继续。

## 5. v0.2 完成标准

- P13/P14/P15 全部合入 main，CI 绿。
- `pnpm test:ci` 测试数从 v0.1 的 464+ 增长，且无回归。
- dogfood code-change workflow 跑一次端到端：
  - `code-map` 与 `risk-scan` 实测并行（wall-clock 下降 ≥30%）。
  - 一次 Claude 子进程超时能被 retry 自动恢复。
  - agent stdout 落 artifact，event log 可单文件复盘整个 run。
  - 在 `review` job 前插入 `human` step，能手动 approve 推进后续 job。
- 发布 v0.2.0 tag + Release Notes（包括迁移指引：新增事件 / artifact kind / 配置字段对默认 workflow 模板的影响）。
