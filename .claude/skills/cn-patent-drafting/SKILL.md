# 中国专利申请文件撰写 Skill

## 总体工作流程

遇到专利相关任务时，**必须**按以下流程操作：

1. **判断专利类型** → 发明专利 / 实用新型专利 / 外观设计专利
2. **收集技术信息** → 向用户提问，填充必要技术细节（见下方问卷）
3. **确定撰写策略** → 确定保护范围策略（宽保护 vs. 稳健保护）
4. **按规范起草文件** → 参照本 Skill 的格式规范逐部分撰写
5. **自检与提示** → 对照审查要点清单自检，提示用户注意事项

---

## 第一步：信息收集问卷

在撰写任何专利文件之前，**必须**向用户确认以下信息（可分批询问，不要一次性抛出所有问题）：

### 基础信息
- [ ] 专利类型（发明 / 实用新型 / 外观设计）
- [ ] 发明名称（简洁、技术性，避免商标名/人名）
- [ ] 发明人姓名 & 申请人（单位/个人）
- [ ] 技术领域（IPC 分类方向，如不清楚可让用户描述应用场景）

### 技术内容（核心）
- [ ] **要解决的技术问题**：现有技术存在什么缺陷/不足？
- [ ] **技术方案**：具体如何解决？关键结构/步骤/参数是什么？
- [ ] **有益效果**：相比现有技术有哪些改进/优势？
- [ ] **实施例**：是否有具体的产品/装置/方法可以描述？
- [ ] **附图**：是否有结构图、流程图、电路图等？

### 保护范围意图
- [ ] 最核心的创新点是什么（独立权利要求的基础）？
- [ ] 有哪些优选/可选的技术特征（从属权利要求的基础）？
- [ ] 是否希望同时申请软件/方法/装置等多维度保护？

---

## 第二步：专利类型判断指引

| 类型 | 客体 | 保护期 | 主要特点 |
|------|------|--------|----------|
| **发明专利** | 产品、方法或其改进 | 20年 | 需实质审查，要求新颖性+创造性+实用性 |
| **实用新型专利** | 产品的形状、构造或结合 | 10年 | 仅初审，不审查创造性，授权快（约6-12月）|
| **外观设计专利** | 产品外观（形状/图案/色彩） | 15年 | 不涉及内部结构，保护视觉设计 |

**判断建议**：
- 软件算法/化学方法 → 发明专利（实用新型不适用于方法）
- 机械结构改进、快速上市需求 → 实用新型（或发明+实用新型同时申请）
- 产品外观造型 → 外观设计专利

## 第三步：文件结构与撰写规范

### 发明专利 / 实用新型专利所需文件
1. [专利请求书]（模板化表格，本 Skill 不生成）
2. **说明书**（必须）← 本 Skill 重点
3. **权利要求书**（必须）← 本 Skill 重点
4. **说明书摘要**（必须）← 本 Skill 重点
5. 说明书附图（有附图时必须）
6. 摘要附图（有附图时必须）

> 详细格式规范见：`references/specification.md`（说明书）、`references/claims.md`（权利要求书）、`references/abstract.md`（摘要）

---

## 第四步：撰写顺序（重要！）

**正确顺序**：权利要求书草稿 → 说明书 → 摘要

理由：权利要求书是专利的"灵魂"，确定保护范围后，说明书需为每项权利要求提供充分支持，摘要最后根据说明书提炼。

---

## 第五步：权利要求书撰写核心原则

### 独立权利要求
- 必须包含实现发明目的的**全部必要技术特征**，缺一不可
- 保护范围应尽量宽泛（删除非必要限定）
- 格式：`一种[名称]，其特征在于，[技术特征1]，[技术特征2]，……`
- 产品权利要求：用结构特征描述
- 方法权利要求：用步骤/工艺参数描述

### 从属权利要求
- 引用在前的权利要求，进一步限定优选方案
- 格式：`根据权利要求X所述的[名称]，其特征在于，[附加特征]`
- 建议层次：独立权 → 第一层从属（核心优选）→ 第二层从属（具体参数）

### 禁忌
- ❌ 不得使用"约"、"大约"、"接近"、"等"等模糊词
- ❌ 不得在权利要求中解释原因/理由（只写特征，不写效果）
- ❌ 独立权利要求不得引用其他权利要求
- ❌ 每项权利要求只在末尾使用一个句号

> 详细规范见：`references/claims.md`

---

## 第六步：说明书撰写结构（五段式）

```
【技术领域】
【背景技术】
【发明内容】
  ├── 技术问题
  ├── 技术方案
  └── 有益效果
【附图说明】（有附图时）
【具体实施方式】
```

每个部分须以对应标题开始，不可省略（外观设计除外）。

> 各部分详细要求见：`references/specification.md`

---

## 第七步：自检清单（撰写完成后必须执行）

### 权利要求书自检
- [ ] 独立权利要求是否包含全部必要技术特征？
- [ ] 独立权利要求是否存在多余限定（可删除的特征）？
- [ ] 从属权利要求引用关系是否清晰？
- [ ] 权利要求用词是否与说明书一致？
- [ ] 是否存在模糊词汇（"约"、"较佳"等）？
- [ ] 方法权利要求的步骤顺序是否必要？

### 说明书自检
- [ ] 背景技术是否分析了现有技术的缺陷（避免过度贬低）？
- [ ] 发明内容是否与权利要求书保持一致？
- [ ] 具体实施方式是否支持最宽独立权利要求？
- [ ] 是否有至少一个优选实施例？
- [ ] 附图标记是否在说明书中首次出现时标注？

### 摘要自检
- [ ] 字数是否在 300 字以内？
- [ ] 是否包含：技术领域 + 技术问题 + 技术方案要点 + 主要用途？
- [ ] 是否避免了宣传性语言？

## 常见驳回意见及应对策略

| 审查意见 | 原因 | 应对策略 |
|---------|------|----------|
| 新颖性缺乏（§22.2） | 被对比文件全文公开 | 补充区别技术特征，或缩小保护范围 |
| 创造性不足（§22.3） | 对比文件+公知常识可推出 | 强调技术效果的意外性；增加技术特征 |
| 说明书公开不充分（§26.3） | 实施例不足，领域技术人员无法实现 | 补充实施例、数据、参数范围 |
| 权利要求缺乏支持（§26.4） | 权利要求范围超出说明书记载 | 缩小权利要求；或补充说明书实施例 |
| 权利要求不清楚（§26.4） | 技术特征表述模糊 | 替换模糊词汇，使用技术精确表述 |

## 输出格式要求

撰写专利文件时，使用以下 Markdown 格式：

```markdown
# [发明名称]

## 权利要求书

**权利要求1**（独立权利要求）
一种[名称]，其特征在于，……

**权利要求2**（从属权利要求）
根据权利要求1所述的[名称]，其特征在于，……

---

## 说明书

**【技术领域】**
……

**【背景技术】**
……

**【发明内容】**
……

**【附图说明】**
……

**【具体实施方式】**
……

---

## 说明书摘要

……
```

输出结束后，附上：
1. 该方案的**主要风险提示**（如现有技术检索建议）
2. **后续步骤建议**（如是否建议同时申请实用新型、PCT路径等）
